# 产品需求文档 (PRD)：智能作业批改系统 (MVP 版)

## 1. 项目概述

### 1.1 产品背景与目标

传统的理工科作业（尤其是涉及大量公式、推导和画图的学科，如《自动控制原理》）批改耗费教师大量精力。本产品旨在通过 **大视觉语言模型 (VLM) + 检索增强生成 (RAG)** 技术，自动化完成作业批改。

**核心目标**：实现“输入图片+标准答案 $\rightarrow$ 输出步骤级得分与知识点反馈”的闭环，不仅判断对错，更要做到“知其所以然”，辅助教学闭环。

### 1.2 目标用户

- **核心用户**：高校/高中的理工科教师及助教。
    
- **延伸用户**：需要针对性反馈以查漏补缺的学生。
    

---

## 2. 核心业务流程

1. **任务创建**：教师上传本次作业的题干、标准答案（支持图片/文本）及得分细则（Rubric）。
    
2. **批量导入**：教师批量上传学生提交的作业解答图片。
    
3. **图像解析 (AI 处理阶段 1)**：系统调用 VLM API，将手写解答过程结构化提取为文本/公式，并识别关键步骤。
    
4. **逻辑比对 (AI 处理阶段 2)**：系统将提取的学生步骤与标准答案步骤进行逻辑对齐与比对。
    
5. **诊断与 RAG 检索 (AI 处理阶段 3)**：
    
    - **若错误**：定位错误步骤，提取相关概念，从《自动控制原理》向量库检索知识点，生成错因分析。
        
    - **若正确**：进行“伪正确”审查（检查步骤是否跳跃、逻辑是否倒推），如有问题则指出；无问题则判定全对。
        
6. **报告生成**：输出最终成绩、得分明细及个性化学情反馈。
    

---

## 3. 功能需求详细说明

### 3.1 任务输入与管理模块

- **标准答案录入**：支持文本输入、Markdown 格式（用于数学公式）或直接上传标答图片交由系统解析。
    
- **评分细则 (Rubric) 配置**：
    
    - 允许教师设定每个步骤的分值（例：列出微分方程得 2 分，拉普拉斯变换 $F(s)$ 得 3 分，求出传递函数得 5 分）。
        
    - 支持设定“容错率”（如：最终计算错误扣 1 分，但方法正确保留步骤分）。
        
- **作业批量上传**：支持 .jpg, .png, .pdf 格式，支持 ZIP 压缩包批量解析，自动识别学号/姓名以绑定学生身份。
    

### 3.2 核心 AI 批改引擎（黑盒逻辑）

这是产品的核心竞争力，分为以下几个子模块：

- **多模态解析提取器**：
    
    - **功能**：精准识别手写中英文、数学公式、专业符号（如控制工程中的积分环节、阻尼比 $\zeta$ 等），并提取为结构化数据。
        
- **智能步骤对齐与比对算法**：
    
    - **功能**：非死板的文本匹配，而是语义和数学逻辑层面的对齐。学生可能使用与标答不同的变量名或等价解法，大模型需具备同义替换和等价公式判断能力。
        
- **错误诊断与 RAG 增效**：
    
    - **功能（纠错）**：一旦判定某步骤逻辑中断或计算错误，立即触发 RAG 检索。
        
    - **RAG 动作**：将错误上下文作为 Query，在已构建的《自动控制原理》教科书向量库中进行相似度检索，提取相关定理（如劳斯稳定判据、根轨迹绘制规则）。
        
    - **功能（查漏）**：针对“答案正确但步骤存疑”的情况，模型需专门通过 prompt 执行逻辑严密性校验，防止学生“凑答案”。
        

### 3.3 结果输出与反馈模块

- **批改结果看板**：教师端呈现批注后的作业图片（在出错位置生成锚点或红框）。
    
- **学生专属诊断报告**：
    
    - **成绩与得分细则**：列出每一步的得分情况。
        
    - **错因剖析**：例如“你在第 3 步绘制伯德图时，未考虑系统包含积分环节导致起始斜率计算错误”。
        
    - **知识点拓展**：结合向量库内容，输出专业知识讲解（如：“建议复习《自动控制原理》第 5 章频域分析法关于开环对数频率特性的内容”）。
        

---

## 4. 技术实现建议与架构

为了让您的产品能稳健落地，建议在技术选型上参考以下架构：

- **视觉与大语言模型 (VLM)**：由于涉及复杂的公式和步骤，建议调用推理能力极强的 API（如 GPT-4o 或 Claude 3.5 Sonnet），它们对理工科手写笔迹和逻辑推导有较好的原生支持。
    
- **RAG 向量库 (Vector Database)**：
    
    - 可以利用 **Supabase** (其 pgvector 插件非常适合快速构建 RAG) 或 Milvus。
        
    - **数据预处理**：需要将《自动控制原理》教科书按章节、知识点、例题进行高质量的 Chunking（分块），并使用合适的 Embedding 模型向量化，确保检索的召回率。
        
- **Prompt Engineering**：
    
    - 需要设计一套复杂的 **Chain-of-Thought (CoT)** Prompt 模板。例如：`"作为自动化专业教授，请先提取学生的解题步骤，然后与给定的标准步骤进行逻辑匹配。不要直接给出结论，请一步步分析学生的推导过程..."`
        

---

## 5. 非功能性需求

- **准确率指标**：图像公式识别准确率需达到 90% 以上；误判率（将对的判错）需严格控制在 5% 以下，宁可提示“需人工复核”，不可乱扣分。
    
- **响应时间**：单份作业（2-3页图片）的解析、比对、RAG 检索和报告生成，整体耗时应控制在 15-30 秒内。
    
- **并发能力**：支持教师在期中/期末等节点进行百张级别的并发任务提交，后端需引入消息队列（如 RabbitMQ/Celery）进行异步处理。
    

## 6. 项目演进路线 (Roadmap)

- **Phase 1 (MVP 版本)**：跑通核心流程。仅支持单一种类的题目（例如纯计算推导题），打通 VLM 识别与教材 RAG 知识点反馈。
    
- **Phase 2 (进阶版本)**：增加对复杂图表的支持（如学生手绘的根轨迹图、奈奎斯特图的比对与打分）；支持多题型、多学科横向扩展。
    
- **Phase 3 (商业化版本)**：沉淀学生学情数据，反向为教师提供“全班共性错误分析报告”；为学生生成“个性化错题本”及知识图谱弱点分析。



# 智能作业批改系统：技术架构与实施指南

## 一、 系统技术架构 (Technology Stack)

基于快速验证 MVP 以及后续的高效扩展，系统采用前后端分离，核心 AI 逻辑微服务化的架构。

### 1. 核心技术栈选型

- **后端框架 (Backend)**：`Python` + `FastAPI`。Python 在 AI 生态中拥有绝对统治力，FastAPI 的异步特性非常适合处理耗时的 VLM API 请求和并发批改任务。
    
- **数据库与向量存储 (Database & Vector Store)**：推荐使用 **Supabase**。
    
    - **关系型数据**：利用其底层的 PostgreSQL 存储教师信息、作业元数据、学生成绩和评分细则 (Rubric)。
        
    - **向量存储**：通过开启 `pgvector` 插件，直接在同一实例中构建《自动控制原理》的 RAG 向量库，极大降低基础架构的维护成本。
        
- **AI 与编排层 (AI Orchestration)**：
    
    - **大模型**：视觉与逻辑推理调用 GPT-4o 或 Claude 3.5 Sonnet API。
        
    - **框架**：`LangChain` 或 `LlamaIndex` 用于串联 VLM 识别、RAG 检索和最终报告生成的 Agent 工作流。
        
- **前端展示 (Frontend)**：MVP 阶段可使用 `Streamlit` 或 `Gradio` 快速搭建交互原型，方便直接处理图片上传和富文本/Markdown 展示；后续商业化转为 Vue/React 结合 Canvas 渲染图片批注框。
    

### 2. 逻辑架构分层

1. **接入层 (API Gateway)**：处理图片批量上传、权限校验、任务分发。
    
2. **业务逻辑层 (Business Logic)**：作业管理、成绩计算、PDF/图片解析。
    
3. **AI 核心引擎 (AI Engine)**：
    
    - **解析器 (Parser)**：接收图片，输出结构化的数学公式（LaTeX 格式）和中文步骤。
        
    - **比对器 (Matcher)**：执行 Chain-of-Thought (CoT)，将学生步骤与标答步骤对齐。
        
    - **诊断器 (Diagnostician)**：触发检索，分析错因。
        
4. **数据存储层 (Data Access)**：Supabase (关系数据表 + `pgvector` 索引)。
    

---

## 二、 核心模块数据流向

以批改一道“求闭环传递函数并判断系统稳定性”的题目为例，数据流向如下：

1. **输入**：学生手写图片 (Image) + 标答 JSON (包含步骤 1、2、3 及对应分值)。
    
2. **多模态提取**：VLM 接收 Image，将其转化为结构化 JSON，例如 `{"step_1": "写出开环传递函数 $G(s)$", "step_2": "列出特征方程 $1+G(s)=0$", "final_answer": "不稳定"}`。
    
3. **逻辑对齐与判分**：后端遍历学生的步骤与标答，发现学生在“列出特征方程”时，符号写错导致最终结果错误。系统在此步骤扣分。
    
4. **RAG 诊断触发**：将出错上下文 `"特征方程符号错误导致劳斯判据判断失误"` 转化为向量 $v \in \mathbb{R}^d$。
    
5. **向量检索**：在 Supabase 的 `pgvector` 中进行余弦相似度检索，召回《自动控制原理》中“劳斯稳定判据”的定义与易错点。
    
6. **综合生成**：大模型结合错误步骤和召回的知识点，生成最终评语：“你的特征方程符号存在计算失误。根据劳斯判据，特征方程各项系数必须大于零系统才可能稳定，请重点复习劳斯表的列写规则。”
    

---

## 三、 详细操作步骤 (MVP 阶段落地指南)

以下是为您梳理的从零开始跑通核心业务流的开发步骤。

### 阶段 1：本地环境与基础设施搭建

1. **初始化开发环境**：
    
    - 创建 Python 虚拟环境，安装核心依赖：`pip install fastapi uvicorn langchain openai supabase psycopg2-binary sentence-transformers`。
        
    - 考虑到您在本地进行开发和初步测试，如果在 MacBook M 系列芯片上运行本地的 Embedding 模型（例如用于快速文本向量化的 `BGE-m3`），可以在 PyTorch/HuggingFace 配置中指定 `device='mps'` 以利用 Apple Silicon 的硬件加速，极大提升处理教材数据的速度。
        
2. **配置 Supabase 数据库**：
    
    - 在 Supabase 控制台新建项目，进入 SQL Editor，开启向量扩展：
        
        SQL
        
        ```
        CREATE EXTENSION IF NOT EXISTS vector;
        ```
        
    - 创建两张核心表：
        
        `textbook_knowledge` (用于 RAG，包含 `id`, `content`, `metadata`, `embedding vector(1024)`)
        
        `homework_records` (用于存储批改结果，包含学生信息、最终得分、生成的 JSON 报告)。
        

### 阶段 2：构建《自动控制原理》RAG 向量库

1. **教材数据预处理 (Data Ingestion)**：
    
    - 将教材 PDF 转为 Markdown 格式。
        
    - **关键点**：由于控制工程涉及强逻辑，不能简单按字数切分 (Chunking)。需按“概念-公式-例题”为基本单元进行语义切片。例如，将“奈奎斯特稳定判据”的定义、公式和一两道例题放在同一个 Chunk 中。
        
2. **向量化与入库**：
    
    - 编写 Python 脚本，调用 Embedding 模型（如 `text-embedding-3-small`）将切片转化为向量。
        
    - 通过 Supabase 的 Python Client，将内容和向量批量 `insert` 到 `textbook_knowledge` 表中。
        

### 阶段 3：开发 AI 批改流水线 (Pipeline)

这一步是整个系统的灵魂，建议分拆为三个连续的 Prompt 节点：

1. **节点 A：视觉解析器 (OCR & Layout Extraction)**
    
    - **操作**：调用 GPT-4o 接口，传入学生作业图片。
        
    - **Prompt 设计**：“你是一个公式与手写识别专家。请将图片中的解答步骤逐一提取，遇到数学公式请严格使用 LaTeX 语法输出。最后以 JSON 数组返回：`[{"step_id": 1, "content": "..."}]`。”
        
2. **节点 B：逻辑裁判引擎 (Logic Matcher)**
    
    - **操作**：将节点 A 提取的 JSON 与教师预设的标准答案 JSON 一并交给模型。
        
    - **Prompt 设计**：“比较学生步骤与标准答案。判断学生的每一步是否逻辑成立。允许同义替换或等价代数变形。如果发现错误，明确指出在第几步、错误类型（计算错/概念错/逻辑断裂）。”
        
3. **节点 C：RAG 增强反馈生成 (Feedback Generator)**
    
    - **操作**：如果节点 B 返回了错误信息，提取错误关键词，连接 Supabase 进行相似度检索（`similarity_search`）。
        
    - 将召回的教材原文连同节点 B 的判断，发送给最终的模型节点，生成带有得分细则和知识点讲解的最终批改报告。
        

### 阶段 4：API 封装与联调测试

1. **暴露接口**：使用 FastAPI 编写 `/upload_homework` 和 `/get_grading_report` 接口。
    
2. **组装测试集**：收集 20-30 份真实的《自动控制原理》手写作业（包含全对的、典型计算错误的、思路完全偏离的）。
    
3. **调优迭代**：
    
    - 观察测试集结果，若发现模型看不懂手绘的伯德图，则需要在节点 A 的 Prompt 中强化对图表坐标轴和关键点（如转折频率）的提取指令。
        
    - 若发现 RAG 召回不准，则需要回去优化教材的 Chunking 策略，增加元数据（如 `chapter: 频域分析`）。
